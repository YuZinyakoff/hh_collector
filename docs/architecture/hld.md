# HLD: Платформа накопления данных о вакансиях hh.ru

**Статус:** Draft v1  
**Назначение:** верхнеуровневый дизайн (High-Level Design)  
**Фокус этапа:** накопление и сохранение данных  
**Целевой режим:** длительный stateful-сбор с историей наблюдений  
**Язык документа:** русский

---

## 1. Контекст

В рамках проекта требуется спроектировать и реализовать платформу длительного накопления данных о вакансиях с hh.ru через публичное API.

На текущем этапе цель проекта — **не исследовательская аналитика как таковая**, а создание надежной, воспроизводимой и расширяемой системы сбора, хранения и контроля качества данных.

Система должна:

- регулярно собирать данные о вакансиях;
- сохранять историю наблюдений;
- быть пригодной для долгосрочной эксплуатации;
- быть удобной для мониторинга и сопровождения;
- позволять позже добавить исследовательский или аналитический слой без перестройки базовой архитектуры.

---

## 2. Цели

### 2.1. Бизнес-/исследовательские цели текущего этапа

1. Построить устойчивую платформу накопления данных о рынке вакансий.
2. Получать воспроизводимый массив наблюдений за длительный период.
3. Сохранять не только "текущие" вакансии, но и историю их видимости во времени.
4. Обеспечить возможность последующего построения панелей и производных витрин.
5. Минимизировать риск потери данных, тихих сбоев и неочевидной деградации качества сбора.

### 2.2. Инженерные цели

1. Реализовать **stateful crawler system**.
2. Разделить сбор, хранение, оркестрацию и наблюдаемость по понятным контурам.
3. Обеспечить идемпотентность и контроль повторных запусков.
4. Сделать систему пригодной для запуска на одной VPS на старте.
5. Подготовить основу для агентской разработки и итеративной имплементации.

---

## 3. Область охвата

## 3.1. Входит в текущий этап

- интеграция с hh API;
- планирование и выполнение обхода;
- сбор списков вакансий;
- выборочный сбор детальных карточек;
- хранение сырых и нормализованных данных;
- хранение истории наблюдений;
- оркестрация запусков;
- логирование, метрики, алертинг;
- резервное копирование;
- подготовка данных для дальнейшего расширения.

## 3.2. Не входит в текущий этап

- финальная исследовательская методология;
- построение эконометрических моделей;
- классификация AI / non-AI;

- сложный BI-слой;
- пользовательский продуктовый интерфейс.

Исследовательский слой должен быть **поддержан архитектурно**, но не является обязательным deliverable текущего этапа.

## 4. Архитектурные принципы

1. **Stateful over stateless**  
   Система должна помнить, что уже видела, когда видела и что изменилось.

2. **Raw first**  
   Сырые ответы API должны сохраняться для воспроизводимости и отладки.

3. **Separation of concerns**  
   Сбор, состояние обхода, хранение, оркестрация и наблюдаемость должны быть разделены логически.

4. **Idempotent ingestion**  
   Повторный запуск не должен ломать данные или приводить к неконтролируемым дублям.

5. **Observability by design**  
   Метрики, логи, ошибки и алерты — часть MVP.

6. **Global sweep as source of truth**  
   Канонический датасет строится из глобального sweep-контура, а не из приоритетных/ускоренных выборок.

7. **Selective detail fetching**  
   Детализация должна быть выборочной, а не тотальной по расписанию.

8. **Single-node friendly**  
   На старте система должна помещаться на одной VPS.

9. **Extensible storage model**  
   Модель данных должна позволять потом добавить enrichment, labels, фичи и агрегаты.

---

## 5. Целевой режим работы

Базовый целевой режим:

- **еженедельный глобальный sweep** поискового пространства hh.ru;
- **state tracking** по вакансиям;
- **выборочная детализация** новых и/или изменившихся вакансий;
- **длительное накопление истории наблюдений**.

Это означает:

- система регулярно обходит пространство поиска;
- фиксирует факт наблюдения вакансии в конкретный момент;
- хранит историю first_seen / last_seen / seen_count / missing_count;
- не делает обязательную полную детализацию всех вакансий на каждом цикле.

## 7. Системы и подсистемы

## 7.1. Ingestion System

Отвечает за взаимодействие с API hh.ru.

### Подсистемы

#### 7.1.1. API Client
Единый клиент для работы с hh API.

Функции:
- формирование запросов;
- добавление обязательных заголовков;
- retry/backoff;
- обработка 429/5xx;
- таймауты;
- сериализация ответов.

#### 7.1.2. Partition Planner
Формирует пространство обхода.

Функции:
- создание tree-based партиций обхода;
- параметризация партиций через disjoint search scopes;
- стартовое exhaustive planning по корневым area partitions;
- адаптивное дробление saturated partitions в child scopes;
- подготовка списка jobs для sweep.

#### 7.1.3. List Crawler
Собирает поисковую выдачу вакансий.

Функции:
- обход страниц выдачи;
- recursive execution по terminal partitions planner v2 tree;
- saturation-aware интерпретация search scope;
- извлечение коротких объектов вакансий;
- сохранение результатов в raw и operational storage;
- генерация seen events;
- постановка detail-jobs при необходимости;
- split saturated parent scopes в child partitions вместо ложного "полного покрытия".

#### 7.1.4. Detail Fetcher
Получает детальные карточки вакансий.

Функции:
- выборочная детализация;
- загрузка полной карточки вакансии;
- запись snapshot;
- запись raw payload.

#### 7.1.5. Dictionary Sync
Обновляет справочники API.

Примеры:
- регионы;
- профессиональные роли;
- типы занятости;
- графики;
- опыт;
- иные справочные сущности API.

#### 7.1.6. Rate Limit Controller
Контролирует интенсивность запросов.

Функции:
- глобальный throttling;
- адаптивное снижение скорости;
- управление конкурентностью;
- backoff на ошибки;
- защита от перегрузки API.

---

## 7.2. Crawl State & Metadata System

Отвечает за "память" системы сбора.

### Подсистемы

#### 7.2.1. Run Registry
Журнал запусков и служебных операций.

Функции:
- регистрация sweep-run;
- статусы запусков;
- время старта/окончания;
- агрегированная статистика по run;
- coverage reporting поверх planner v2 tree без прямого SQL.

#### 7.2.2. Partition State Store
Хранит состояние каждой партиции.

Функции:
- pending / running / split_required / split_done / done / failed / unresolved;
- количество страниц;
- количество найденных вакансий;
- число ошибок;
- контроль повторных попыток;
- parent/child lineage и coverage semantics для exhaustive crawl.

Для planner v2 partition state должен уметь различать:

- execution lifecycle конкретной партиции;
- coverage outcome партиции как search scope;
- факт saturation и последующего split.

С точки зрения покрытия дерева:

- terminal leaf со статусом `done` и `coverage_status=covered` означает, что данный search scope дочитан до конца;
- saturated parent со статусом `split_done` и `coverage_status=split` означает, что сам scope не считается покрытым напрямую и делегирует покрытие child scopes;
- `unresolved` означает, что покрытие этого scope не было достигнуто и refinement по текущей split-policy не удался.

Operator-facing reporting для `crawl_run` должен уметь ответить без SQL:

- сколько terminal leaves уже `covered`;
- сколько leaves всё ещё `pending` или `running`;
- сколько parent nodes находятся в `split`;
- есть ли `failed` или `unresolved` узлы;
- как выглядит текущее partition tree данного run.

#### 7.2.3. Vacancy Seen-State Tracker
Хранит историю наблюдений вакансий.

Функции:
- first_seen_at;
- last_seen_at;
- seen_count;
- consecutive_missing_runs;
- статус "активно наблюдается / предположительно исчезла".

#### 7.2.4. Idempotency & Dedup Layer
Защищает от дублей и повторной записи.

Функции:
- идемпотентные ключи;
- контроль уникальности событий;
- dedup вакансий, snapshots и raw записей.

## 7.4. Processing System

Отвечает за постобработку после сбора.

На текущем этапе это **техническая обработка**, а не исследовательская аналитика.

### Подсистемы

#### 7.4.1. Normalization Pipeline
Нормализует данные из API.

Функции:
- приведение схем;
- очистка текстов;
- унификация nullable-полей;
- приведение справочных ссылок к внутренним идентификаторам.

#### 7.4.2. Snapshot Builder
Формирует исторические snapshots.

Функции:
- запись полных состояний вакансии;
- фиксация времени наблюдения;
- фиксация изменений между версиями.

#### 7.4.3. Change Detection
Определяет, изменилось ли состояние объекта.

Функции:
- hash short payload;
- hash detail payload;
- сигнал "изменилось / не изменилось";
- принятие решения о повторной детализации.

## 7.5. Orchestration & Control Plane

Отвечает за управление системой.

### Подсистемы

#### 7.5.1. Scheduler
Планирует регулярные задачи.

Примеры задач:
- weekly sweep;
- dictionary sync;
- retention jobs;
- backup jobs;
- health jobs.

#### 7.5.2. Job Queue
Очередь задач для воркеров.

Типы job:
- list crawl;
- detail fetch;
- retry crawl;
- reprocessing;
- maintenance.

#### 7.5.3. Worker Pool
Исполняет задачи из очереди.

Функции:
- параллельное выполнение;
- масштабирование по типам задач;
- изоляция разных классов задач.

#### 7.5.4. Admin / Ops Interface
Минимальный операционный интерфейс.

Формат:
- CLI;
- management commands;
- служебные скрипты;
- при необходимости — простой внутренний web/admin.

Функции:
- pause/resume;
- rerun partition;
- replay failed jobs;
- inspect run;
- force detail refresh.
- execute `run-once` для legacy smoke orchestration-lite path;
- execute `run-once-v2` для tree-aware exhaustive collection path без ручной сборки planner/list/detail/reconcile шагов.
- execute `trigger-run-now` для одного guarded запуска tree-aware orchestration v2;
- execute `scheduler-loop` для unattended interval-based execution без queue/distributed orchestration.

`run-once-v2` должен оркестрировать уже существующие use cases, а не дублировать их логику:

- создать `crawl_run`;
- выполнить initial planning через planner v2;
- повторять tree-aware list execution до одного из честных terminal outcomes;
- запускать selective detail stage только после полного list coverage;
- завершать run через reconciliation;
- возвращать operator-friendly итоговый статус.

Для текущего этапа stop condition формулируется через coverage semantics дерева:

- success: `coverage_ratio = 1.0`, `pending_terminal_partitions = 0`, `unresolved_partitions = 0`, `failed_partitions = 0`;
- `completed_with_detail_errors`: list coverage завершён полностью, но selective detail stage вернул failed fetches;
- `completed_with_unresolved`: failed partitions нет, но есть unresolved leaves/scopes;
- failed: есть failed partitions или критическая orchestration/list ошибка.

## 8. Потоки данных

## 8.1. Основной сценарий weekly sweep

1. Scheduler запускает новый `CrawlRun`.
2. Partition Planner формирует набор `CrawlPartition`.
3. Для каждой партиции в очередь ставится list-crawl job.
4. List Crawler получает страницу выдачи через API.
5. Raw payload сохраняется в Raw Storage.
6. Из выдачи извлекаются краткие данные по вакансиям.
7. Для каждой вакансии:
   - пишется seen-event;
   - обновляется current state;
   - при необходимости создаётся detail job.
8. Detail Fetcher получает полную карточку.
9. Raw detail payload сохраняется.
10. Snapshot Builder пишет snapshot вакансии.
11. После завершения всех партиций `CrawlRun` получает финальный статус.
12. Observability-контур фиксирует метрики, логи и возможные ошибки.

---

## 8.2. Сценарий выборочной детализации

Detail fetch выполняется только если:

- вакансия наблюдается впервые;
- изменился short payload;
- отсутствует актуальный detail snapshot;
- наступил TTL на повторную детализацию;
- выполняется ручной re-fetch по операторской команде.

---

## 8.3. Сценарий исчезновения вакансии

Если вакансия перестала наблюдаться в sweep:

- она не удаляется;
- `last_seen_at` остаётся на последнем моменте наблюдения;
- увеличивается `consecutive_missing_runs`;
- после порогового количества пропусков вакансия помечается как `probably_inactive`.

## 10. Выбор стратегии детализации

## 10.1. Почему не full detail recrawl всего рынка

Полный повторный запрос всех detail-карточек на каждом цикле:
- дорог по API;
- даёт мало пользы на текущем этапе;
- не является обязательным для построения истории наблюдений;
- усложняет эксплуатацию без пропорционального выигрыша.

## 10.2. Почему выбран selective detail fetching

Выборочная детализация:
- сохраняет важные данные о содержимом вакансии;
- уменьшает стоимость обхода;
- позволяет накапливать снапшоты изменений;
- хорошо сочетается с weekly sweep.

## 11. Нефункциональные требования

## 11.1. Надёжность

Система должна:
- переживать временные ошибки API;
- поддерживать retries с backoff;
- не терять run metadata;
- корректно восстанавливаться после рестарта;
- поддерживать перезапуск отдельных партиций.

## 11.2. Наблюдаемость

Система должна:
- экспортировать метрики;
- вести структурированные логи;
- иметь алерты на критичные отклонения;
- позволять расследовать ошибки по run/job ids.

## 11.3. Производительность

Система должна:
- поддерживать долгосрочный сбор на одной VPS стартовой конфигурации;
- иметь управляемую конкурентность;
- не требовать ручного управления каждым запуском;
- поддерживать gradual scale-up.

## 11.4. Идемпотентность

Повторный запуск job не должен приводить к:
- потере состояния;
- неконтролируемым дублям;
- разрушению истории наблюдений.

## 11.5. Расширяемость

Архитектура должна позволять без перелома добавить:
- enrichment pipelines;
- labeling;
- аналитические витрины;
- дополнительные источники данных.

## 11.6. Эксплуатационная простота

На старте система должна:
- устанавливаться и запускаться через Docker Compose;
- иметь понятный конфиг;
- иметь воспроизводимый деплой;
- не требовать Kubernetes.

## 13. Риски

## 13.1. Изменение поведения API
Риск:
- изменение структуры ответа;
- изменение правил rate limiting;
- нестабильность отдельных endpoint.

Смягчение:
- raw storage;
- versioned parsing;
- contract tests;
- throttling и retry policy.

## 13.2. Тихая деградация сбора
Риск:
- sweep формально идёт, но покрытие падает;
- партиции обрабатываются не полностью;
- часть вакансий перестаёт попадать в выборку.

Смягчение:
- метрики покрытия;
- expected vs observed counts;
- алерты на аномалии.

## 13.3. Переполнение хранилища
Риск:
- рост raw JSON;
- рост snapshots;
- нехватка места на VPS.

Смягчение:
- retention;
- архивирование;
- мониторинг диска;
- регулярные бэкапы и очистка.

## 13.4. Потеря истории из-за неправильного dedup
Риск:
- агрессивный dedup уничтожит полезные изменения.

Смягчение:
- разделять seen events и snapshots;
- не смешивать current state и historical state;
- использовать аккуратные правила hash/change detection.

## 13.5. Ошибки оркестрации
Риск:
- run завис;
- очередь остановилась;
- retry storms.

Смягчение:
- watchdog checks;
- лимиты retry;
- ручные operator commands;
- алерты.
